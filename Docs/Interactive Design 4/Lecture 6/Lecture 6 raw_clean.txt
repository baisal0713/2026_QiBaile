## Review: Bodies in Space and the Interaction Model

Let me review what we've been talking about. The theme we started two lectures ago and are concluding today is this idea of bodies in space -- how just through the use of the body, presence, absence, and today we also introduce gaze, we can create interactions. We gave some examples with flowers that illuminate, Rain Room, and perhaps today we'll also see something with particles.

We said we can have presence without gaze, or also contact. Contact we saw last semester too -- the simple trigger. From next time we start actually building interactions in various modalities that are a step forward from these things. But the most important thing we introduced last time, and now examine a bit more closely, is this basic model of interaction -- which actually doesn't belong to interaction design per se. It comes from signal processing, data processing: the simple model of input, process, and output.

What are input and output for us? As we said last time, we have various components in Unity that we already know -- lights, sounds, particles, post-processing. These components have properties: intensity, volume, and various other things we've seen. This we can call our output -- in an interaction, it will be the feedback.

What can we use as input? Many things. We can use proximity, gaze, as we were saying, stillness -- now I'll show you an example of that which is a bit counterintuitive but actually quite nice. We can also use contact, grab, or even UI buttons. We're not using much UI right now, but keep in mind that as input we can also use time, random sequences -- sometimes we simply need things to change a little in a random sequence.

So what is the logic of what I'd called the atom -- the minimum indivisible part of the interaction? We can combine input and output in multiple ways, including one-to-many relationships and so on. The interesting part in all of this is not simply combining variables -- it's what happens in the middle: the process, the transformation.

## The Four Categories of Interaction

Why am I making these examples? Because I want to talk about these four categories of interaction. Some we know from last semester, others we've seen now, and the most important part -- if you actually want to use interaction design to convey emotions, to create interesting experiences -- is how we design the transformation in the middle.

What does the transformation in the middle mean? We have some parameters that will be more or less always present. The mapping curve we saw last time can be linear or have various shapes -- and that already says a lot. We can have a range, and we can have this thing called an envelope, which I think you've seen from signal processing, like in MSP or Max. If you've worked a bit with synthesizers, the envelope would be attack, release, duration, and sustain. We don't have a continuous wave to shape like in synthesizers, but we have the curve, which is a way in which two or more variables are mapped onto each other.

Now let me show you the way I personally work, which I believe is the best way to also test quickly. The thing I'd like you to remember today -- I'm almost done with the theory -- is these four categories of interaction. What are they based on? They're based on the type of binding and the type of signal.

If we go back a bit, in the first lectures we made a passage where last semester we talked about triggers. This semester we started talking about sensors. What's the difference between a trigger and a sensor? A trigger is basically a thing that fires once -- one-shot. You open the door, something happens. It can obviously trigger an animation, something that has a shape, a behavior, but you saw the problem yourselves: in your project you had to look at a specific object and then do something on that specific object. With a trigger, that doesn't work. For that you need a sensor -- the only difference being that the sensor sends a signal. The signal can be empty if we don't have particular requirements, but if we need to identify a specific object, a specific position in space, the signal can carry this data.

So if we imagine these two axes -- the type of binding and the type of signal -- we get four categories:

**Continuous binding + continuous signal = Tracking.** We take a variable like proximity and map it directly to one or more output variables like intensity. One variable traces the other -- when one moves, the other moves too. That's the continuous interaction.

**Continuous binding + binary signal = Switch.** Similar, but the signal is simply on/off. Enter the room, it turns on; leave, it turns off. We still have a continuous relationship, but the signal isn't continuous -- there's no continuous tracking.

**Unbound + binary signal = One-shot.** We saw this extensively last semester. A single interaction that then plays out its behavior, its curve.

**Unbound + continuous signal = Threshold.** A signal transmits information -- like "my proximity is 1, 2, 3, 5" -- and when it reaches five, something happens. Two variables that aren't continuously paired, but one variable listens to the other's value and when it reaches a certain value, something triggers.

This is a bit abstract and we'll see it in various examples going forward. But if you remember this 2x2 matrix, it helps you think: "If I want something to happen when the player enters a space -- should this be one-shot? Binary? Do I want it continuous?" You design it accordingly, and then it's much easier to implement and test.

## Inverting Interactions

A trick for the transformation is that it can be invertible -- you can reverse the direction. For example, in the stillness dimmer interaction, if you go to the ScriptableObject and look at the values for when it's "lit" versus "dim," by inverting these values we can make it so that when we move, the space becomes dark, and when we're still, it illuminates. This is also a possibility of this approach that separates the data from the controller.

For example, if I go to the stillness dimmer profile, create a new profile called "Stillness Dim Inverse" or "Reverse," and swap the emission lit/dim values and the ambient/reflection values -- keeping the same curves -- and then assign this inverted profile to the Stillness Dimmer, we can test both possibilities with a single controller.

Be aware that the stillness dimmer doesn't activate until the entry animation has completed. Once the entry animation finishes, it activates -- and now we see that if we stand still and then walk, it goes dark. So this inversion is possible: alongside the other transformation parameters, we simply add values for each state of the interaction.

## Building a New Interaction: Compacting Corridor

Now I want to build a new interaction to review the process using ScriptableObjects, and then I'll let you try to design and implement your own interaction, or extend this one.

First, two maintenance things:

**Camera clipping in tight spaces.** Cinemachine has collision functionality, but because these walls are very thin, what happens is the camera can see through them. The solution: go to the Cinemachine camera, find Third Person Follow at the bottom, and check the collision filter -- set it to Default so the camera collides with all colliders on the default layer. Also increase the camera distance a bit -- even to 1 -- so the player's own collider acts as a buffer. Set Avoid Obstacles to included, and lower the camera radius slightly. Playing with these parameters helps reduce that unpleasant clipping effect.

**Light leaking through modular walls.** When working with modular packs, the modules sometimes don't fit perfectly, and light from the directional light enters through the cracks. A common solution is to place a plane covering the entire ceiling, so external light and ambient effects don't leak through.

Now let's build the compacting corridor interaction. Walking through a corridor, the walls gradually close in -- but you can still squeeze through. What type of interaction is this?

[A student suggests it's continuous/tracking.]

Right. Based on the player's position in the corridor -- if you're at the entrance, you're fine; at the midpoint, the walls have moved; at the end, they're just barely passable. So we'd have continuous binding, continuous signal -- a tracking interaction using proximity/distance.

### The Two-Pass Design Approach

I recommend separating the design of the interaction from the implementation. We work in two passes: first we design the interaction -- write it down, either by hand or with AI help -- and in the second pass we take the spec and implement it. We can also ask AI to help expand the spec.

I've written just two sentences for the compacting corridor spec. I can open an AI agent, give it this file with my two sentences, and ask it to help me design this interaction. It's useful for two reasons: the AI will fill in a more complete specification with properties, and we can read the spec in human language to validate the interaction design rather than reading code.

### Implementing the Corridor

I'll implement it and pass it to you. We need to think about the structure: the modular walls are made of different segments, so I select the wall segments, group them into two parent objects (Wall 1 and Wall 2) using Create Empty Parent. We place a proximity sensor near the exit and set the radius to cover the corridor length. The controller needs: sensor reference, Wall 1, Wall 2, the movement axis, and the profile.

Create the profile asset: right-click, Create > LudCore > Compacting Corridor Profile. Set the max distance to match the corridor length (about 15), max compaction to around 1-1.2, start with a linear curve, and leave attack/release speed at zero for now.

## Adding Feedback Layers to the Compacting Corridor

Now let me show you how to add a couple more feedback outputs to this interaction.

First, **material animation**: on the wall material, we can animate the texture offset. If the walls have arrow textures, we can make the arrows scroll downward -- and the scroll speed increases with proximity, reinforcing the sense of urgency.

Second, **post-processing**: we can use vignette and desaturation. Adding a vignette creates a tunnel-vision effect, and desaturating the color reinforces the claustrophobic feeling. Both mapped to proximity.

I update the spec with these additional feedbacks and ask the AI to update the implementation. I need to create a duplicate material specifically for the corridor walls so we don't modify the global shared material.

When working with materials, we need to specify the correct shader property names. In URP Lit materials, the texture property is called "BaseMap" not "MainTex" -- this is the kind of detail where being precise in the prompt matters.

## Physics: Opening Doors with Hinge Joints

A quick useful thing: how to make doors open with physics. It's actually super simple -- you need a joint.

On the door object, add a Hinge Joint. It will also add a Rigidbody. The key thing is setting the anchor correctly -- the hinge point should be at the edge of the door where it connects to the frame. Set the axis to Y for a standard door rotation. Test by rotating to confirm the correct axis.

Be careful about the player capsule radius -- if it's too large, the player won't fit through doors.

Also important: on the player controller, enable "Can Push" for physics effects, so the player can actually push the door open.

Watch out for door size relative to the door frame colliders -- the door needs to be slightly smaller than the frame, otherwise it gets stuck on the surrounding colliders.
